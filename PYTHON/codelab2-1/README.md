# [Codelab 1] 병렬 데이터 마스킹 및 텍스트 정규화(Normalization)

## 요구사항

- 가이드
  - 수백만 건의 사용자 리뷰 데이터 내에 포함된 이메일 주소, 전화번호 등의 개인정보를 마스킹(`*`) 처리하고, 특정 금칙어를 표준 표현으로 순화하는 배치 프로세서를 구축
  - 단일 프로세스와 멀티 프로세스의 처리량(Throughput)을 비교하여 성능 차이를 분석
  - 데이터가 너무 작을 때 IPC(프로세스 간 통신) 비용으로 인해 멀티 프로세스가 오히려 느려지는 현상을 이해
- 라이브러리
  - `import re`
  - `import multiprocessing as mp`
  - `import csv`
  - `from dataclasses import dataclass`

## 핵심 개념

### 정규표현식(Regex) 오버헤드

복잡한 정규표현식은 CPU 자원을 많이 소모한다. 이를 멀티 프로세스로 분산했을 때 성능이 얼마나 향상되는지 확인하는 것이 이 실습의 핵심이다.

- `re.compile()` : 패턴을 미리 컴파일해두면 반복 호출 시 재해석 비용을 줄일 수 있음
- `pattern.sub(repl, text)` : 매칭된 문자열을 `repl`로 치환. `repl`에 람다 함수를 넘기면 캡처 그룹을 활용한 동적 치환 가능

### 멀티프로세싱(Multiprocessing)과 데이터 청킹(Chunking)

전체 데이터를 CPU 코어 수만큼 청크(chunk)로 나눠 병렬 처리하면 처리량이 향상된다.

- `mp.cpu_count()` : 현재 시스템의 CPU 코어 수 반환
- `mp.Pool(n)` : n개의 워커 프로세스로 구성된 프로세스 풀 생성
- `pool.map(func, chunks)` : 각 청크를 워커에 분배하여 병렬 실행 후 결과를 리스트로 반환

### IPC(Inter-Process Communication) 비용

멀티 프로세스는 데이터를 워커에 전달할 때 직렬화(pickle)·역직렬화 과정을 거친다. 데이터가 너무 작으면 이 통신 비용이 실제 처리 시간보다 커져 단일 프로세스보다 느려진다.

## 마스킹 규칙

| 항목 | 원본 예시 | 마스킹 결과 |
|------|-----------|-------------|
| 이름 | `장아현` | `장*현` |
| 이메일 | `abc123@example.com` | `******@example.com` |
| 전화번호 | `010-1234-5678` | `010-****-****` |
| 리뷰 속 전화번호 | `...053-510-7199에 문의함` | `...053-***-****에 문의함` |

- 이름: 첫 글자·끝 글자 유지, 가운데만 `*` 처리
- 이메일: `@` 앞 로컬파트만 글자 수만큼 `*` 처리
- 전화번호: 앞 번호(지역번호) 유지, 가운데·끝 구간만 `*` 처리

## 실행 결과

```
CSV 로딩 중...
1,000,000건 로드 완료

[단일 프로세스] 처리 중...
  처리 시간: 3.48초
  처리량  : 287,109 rows/s
  저장 완료: masked_single.csv

[멀티 프로세스 - 10코어] 처리 중...
  처리 시간: 2.35초
  처리량  : 424,690 rows/s
  저장 완료: masked_parallel.csv

========================================
단일 프로세스 처리량:      287,109 rows/s
멀티 프로세스 처리량:      424,690 rows/s
속도 향상(speedup)  : 1.48x
========================================

[IPC 비용 관찰 - 소량 데이터 1,000건]
  단일 프로세스: 0.0038초
  멀티 프로세스: 0.0986초
  → IPC 비용으로 인해 멀티프로세스가 오히려 더 느리다
```

- 100만 건 기준 멀티 프로세스(10코어)가 단일 프로세스보다 **1.48배 빠름**
- 1,000건 소량 데이터에서는 멀티 프로세스가 단일보다 **약 26배 느림** → IPC 비용 역전 현상 확인
